# approval-service design

## Scope and business rules

The service owns only the approval workflow. Workspaces, users, publications, scenarios and
edits remain external; the service stores their opaque identifiers and does not call adjacent
services.

A request begins in `pending`. The first successful approve, reject or cancel command moves it
to one terminal state. No transition out of a terminal state is valid. Because the task does
not define quorum semantics, reviewers use an **any-of** rule: any one assigned reviewer with
`approval:decide` can approve or reject. A user with `approval:cancel` can cancel a pending
request in the authenticated workspace.

## Data model

| Table | Purpose |
|---|---|
| `approval_requests` | Aggregate root: source reference, public content fields, status, final decision, actor, timestamps and optimistic version |
| `approval_reviewers` | Ordered external reviewer IDs; unique per request |
| `audit_events` | Append-only business history with actor, action and before/after state |
| `idempotency_records` | Workspace-scoped key, actor, canonical request hash and original HTTP result |
| `outbox_events` | Versioned integration events written atomically with aggregate changes |

UUIDs are generated by the service. Workspace and external IDs are strings so neighboring
services may retain their own identifier formats. Database checks protect status/source enums
and non-negative counters in addition to API validation.

Decision comment/reason is stored on the aggregate because it is part of the public business
result. Audit and outbox records contain only allowlisted metadata; they record whether a note
was supplied but do not copy the note.

## Tenant boundary and authorization

The URL workspace must exactly match `X-Workspace-Id`; otherwise the request fails before a
repository query. Every aggregate and idempotency lookup includes `workspace_id`. A valid
request ID from another workspace therefore returns the same `404` as an unknown ID and does
not disclose existence.

The local header auth is a replaceable boundary. Production integration should validate a
signed service identity at a gateway/middleware and inject the same workspace/user/actions
claims. No credentials belong in this service's public request schemas.

## Idempotency and concurrent decisions

Clients provide `Idempotency-Key` on every mutation. The service hashes a canonical JSON
representation and stores it with an operation identity. A unique constraint on
`(workspace_id, idempotency_key)` is the final concurrency guard.
The recorded actor must also match before a response can be replayed, preventing a key from
being used to retrieve another user's mutation result.

Create, audit, outbox and idempotency rows share one transaction. If two creates race, the
loser rolls back its aggregate on the unique-key conflict and returns the committed response.
For decisions, an atomic conditional update uses
`WHERE workspace_id = ? AND id = ? AND status = 'pending'`. Consequently only one conflicting
terminal decision can commit. A concurrent retry rechecks the idempotency row and replays the
winner when operation and hash match.

The original response is retained so a retry does not depend on later representation changes.
A production retention job should delete records only after the agreed client retry window.

## Audit and integration events

Every successful business mutation appends an `audit_events` row containing:

- workspace and request identifiers;
- actor user identifier;
- action and previous/new state;
- safe, structured metadata;
- occurrence time.

The transactional outbox prevents a database change from committing without its integration
event. Event names are versioned (`approval_request.created.v1`, `.approved.v1`, `.rejected.v1`,
`.cancelled.v1`). Payloads contain only request/workspace/actor IDs, status and time. A future
publisher can claim unpublished rows, send them to a broker, then set `published_at` and
increment `attempts` on failures. Consumers must still deduplicate by event `id` because
outbox delivery is normally at least once.

## Sensitive-data controls

Public request models forbid extra fields and accept only opaque external IDs. Text validation
rejects emails, provider/storage URLs and common token/secret patterns. The validation-error
handler removes Pydantic's submitted `input` before responding. Unexpected-error logs contain
neither exception messages nor request data.

Audit and outbox construction is allowlist-based rather than redaction-based. This avoids
depending on an incomplete secret detector for durable events. Raw provider payloads are not
modeled or persisted.

## Health and migrations

`/health` is process liveness and does not touch dependencies. `/ready` executes `SELECT 1`
against the configured database and returns `503` when unavailable. Schema setup is explicit
through Alembic; the application never calls `create_all` in production.

## Known trade-offs

- The auth mechanism is intentionally a local stub and must be replaced before deployment.
- The any-of reviewer rule is an explicit interpretation; unanimous or ordered approval would
  require per-reviewer decision rows and a quorum policy.
- The outbox table is present, but a broker publisher is outside this test task.
- Idempotency rows have no TTL/cleanup worker in this small service.
- Pagination is offset-based. Cursor pagination would be preferable for very large workspaces.
- SQLite is convenient locally but has coarser write locking; PostgreSQL is the intended
  multi-instance runtime and is the Compose default.
- There is one initial migration. Later schema changes should be additive Alembic revisions.
